# Security Review & Risk Analysis

## 1. Authentication & Authorization Risks
- **Risk**: Unauthorized access via brute-force attacks or leaked tokens.
- **Mitigation**: 
  - Enforce strong password policies and bcrypt hashing.
  - Implement rate limiting on sensitive endpoints (e.g., login, password reset) using `express-rate-limit`.
  - Use short-lived JWTs. For production, the frontend team should store tokens securely (HTTP-only cookies preferred, or secure memory).
  - Strictly enforce Role-Based Access Control (RBAC) via middleware so Customers cannot access Admin endpoints.

## 2. External Integration Risks (Razorpay & Calendly)
- **Risk**: Forged Webhooks (A malicious actor simulating a successful payment or booking webhook).
- **Mitigation**: The backend MUST verify the cryptographic signatures provided by both Razorpay (`x-razorpay-signature`) and Calendly. Unverified webhooks will be immediately rejected with a 400 Bad Request.
- **Risk**: Payment tampering (Frontend manipulating the requested price).
- **Mitigation**: The frontend will never dictate the payment amount. The backend will fetch the service price directly from the database based on the requested Service ID when creating the Razorpay Order.

## 3. Data Validation & Injection Risks
- **Risk**: SQL Injection / NoSQL Injection.
- **Mitigation**: Use Prisma ORM which automatically escapes and uses parameterized queries to interact with the database.
- **Risk**: Invalid input payloads causing crashes or unexpected behavior.
- **Mitigation**: Enforce strict schema validation on all incoming request bodies, query parameters, and URL parameters using a library like **Zod**.

## 4. General API Security Best Practices
- Use `helmet` middleware to set standard secure HTTP headers (XSS filtering, HSTS, no-sniff, etc.).
- Configure CORS (`cors` package) to strictly allow only the intended production/staging frontend domains.
- Ensure that stack traces, detailed error messages, or environment variables are never exposed in production API responses.

## 5. External Credentials Needed Later
To deploy the backend to staging/production or test external integrations, the following credentials must be generated by the project owner and provided in the environment (e.g., `.env` file):

- **Database**:
  - `DATABASE_URL`: Connection string for a PostgreSQL instance.
- **Authentication**:
  - `JWT_SECRET`: A long, random cryptographic string for signing tokens.
- **Razorpay**:
  - `RAZORPAY_KEY_ID`: From the Razorpay developer dashboard.
  - `RAZORPAY_KEY_SECRET`: From the Razorpay developer dashboard.
  - `RAZORPAY_WEBHOOK_SECRET`: Used to verify incoming webhook payloads.
- **Calendly**:
  - `CALENDLY_PERSONAL_TOKEN`: To interact with the Calendly API.
  - `CALENDLY_WEBHOOK_SIGNING_KEY`: Used to verify incoming Calendly webhook payloads.

## 6. Genuine Blockers
- **None at this stage.** 
- The backend architecture is completely decoupled from the frontend. We can build, test, and validate the entire API structure using a local PostgreSQL database, Razorpay test mode, Calendly sandbox/developer environment, and automated tests / Postman.
